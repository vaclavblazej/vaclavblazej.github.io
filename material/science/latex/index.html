Scientific Latex

Here I gather usefull tips and tricks for writing scientific papers in LaTeX. Although I work in theoretical computer science (which has big influence on styles we use) this post may be useful for anyone writing papers in latex.

General

Technique

Given an issue this post shall mention one solution to it eventhough there may exist many different ways to solve the issue. I try to do things so that the process of writing is easier and mistakes are harder to do.

Tools

Compilation

The latex source which contains bibliography can be compiled with the following command.

pdflatex main.tex && pdflatex main.tex && bibtex main.aux && pdflatex main.tex

Editor

  • Texmaker ↗ is a usable multiplatform latex editor.

Newlines

How the code is formatted has its advantages. Two main styles are

  • one paragraph on one line,
  • one sentence on one line,
  • newline when the line is too long.

A paragraph on a line has advantage, that the code looks like the final result.

I prefer one sentence on a line as

  • it makes semantically sense,
  • one can easily see if there are lines which are too long and should to be split into several sentences,
  • working with sentences (e.g. reordering them) becomes easy,
  • it is easy to comment with \todo between sentences,
  • as git versioning system sees differences per line the sentence-per line formatting helps to avoid merge conflicts, and it is easier to see what changed with diff.
  • Newlines to fit the text to the screen is not great because desired width depends on the current editor width.
  • One may setup soft-wrap in the editor to avoid lines getting too long to fit the window.
  • Adding any text to the line which makes it too long breakes the nice newlines.
  • Working with sentences is cubersome as moving it keeps the newlines which will break the nice code formatting.

Indentation

Versioning

You should always backup the project somewhere to ensure that we may not loose the paper progress if something (like PC) breaks. Another consideration whether you collaborate on the paper with someone. We can recommend Git, Dropbox, or Overleaf – each has its advantages and disadvantages.

Git

  • advantages
    • backup in a remote repository
    • the remote repository may be shared with coauthors
    • provides full history of changes (commits)
    • people may edit the same file as long as the one which commits later merges the changes (needed only if the same lines were edited)
    • files are local, editing offline is possible
  • disadvantages
    • (each coauthor) needs some technical skill to operate it

Dropbox

  • advantages
    • backup in your dropbox folder
    • may share the folder with coauthors
    • files are local, editing offline is possible (but may result in problems when dropbox finally synchronizes)
  • disadvantages
    • multiple people may not edit the same file

Overleaf

  • advantages
    • backup in the overleaf project
    • may share the project with cotauthors
    • multiple people may edit the same file (no merging)
  • disadvantages
    • editing can be done only when online
    • more than 2 coauthors is for payed plan only
    • may be usually accessed only via web editor

Custom labels

Taken from the stack exchange answer of Henrik Bøgelund Lavstsen ↗, the answer depends on if you use the hyperref package. If you use hyperref package, then use:

\makeatletter
\newcommand{\customlabel}[2]{%
   \protected@write \@auxout {}{\string \newlabel {#1}{{#2}{\thepage}{#2}{#1}{}} }%
   \hypertarget{#1}{#2}
}
\makeatother

If you do not use hyperref package:

\makeatletter
\newcommand{\customlabel}[2]{%
\protected@write \@auxout {}{\string \newlabel {#1}{{#2}{}}}}
\makeatother
